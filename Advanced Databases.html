📖 WHY OBJECT-RELATIONAL?

Motivation

Relational ✅: Simplicity · Theoretical support · Declarative SQL · Data independence · Optimization · Integrity constraints.
Object ✅: Complex types (multimedia, spatial) · Encapsulation · Inheritance & polymorphism · No excessive joins.
ORDBMS 🔀: Keeps relational robustness + SQL core, adds OO extensions: UDTs, UDFs, Inheritance, OIDs. Standard: SQL-99 / SQL3. Supported by IBM, Oracle, Microsoft, Informix, Sybase.

🧱 ABSTRACT DATA TYPES

ADT / UDT

ADT = composite type grouping attributes + methods. Replaces "class" concept in SQL3. Can contain: member methods (on instance), static methods (on type), constructor methods.
CREATE OR REPLACE TYPE TAddress AS OBJECT (num NUMBER, street VARCHAR2(50), city VARCHAR2(25), zipcode NUMBER);
CREATE TYPE: fails if exists. CREATE OR REPLACE TYPE: replaces if exists, creates if not.
UDTMember methodStatic methodConstructor

📋 TABLE TYPES

Relational vs NF² vs Object

Relational table: classic, simple data, no ADT.
NF² table: Not First Normal Form — has ADT columns but NO OID.
Object table: CREATE TABLE t OF TypeName; — system-generated OIDs, fully polymorphic.
ALTER TYPE ... CASCADE propagates changes to all dependent types/tables. Cannot DROP a type used by a table — ALTER instead.

🔍 SELECT / DML

Dot Notation & TABLE()

Access ADT attr: requires correlation variable + dot path.
SELECT P.NAME, P.ADDRESS.CITY FROM PEOPLE P;
Index on ADT attr (no alias needed):
CREATE INDEX idx ON PEOPLE(ADDRESS.CITY);
TABLE(): flattens VARRAY or NESTED TABLE into queryable rows.
SELECT nt.CITY FROM PEOPLE P, TABLE(P.ADDRESSES) nt;

🔗 OID, REF, DEREF

Object Identity

OID: unique, immutable, system-generated per row in object table.
REF(alias): returns OID pointer. Only on object tables.
DEREF(ref_col): dereferences OID → returns the actual object.
DANGLING REF: REF points to deleted object. Fix: UPDATE t SET col=NULL WHERE col IS DANGLING;
SCOPE IS tableName: restricts REF to one specific object table.
VALUE(alias): returns full object instance (not OID). Used with object tables + views.
Forward declaration: CREATE TYPE name; — breaks circular REF dependency.

⚙️ UDFs & METHODS

User-Defined Functions

MEMBER: operates on instance via SELF. Default type.
STATIC: called on the type: TypeName.method().
MAP MEMBER: maps object to scalar → enables ORDER BY.
ORDER MEMBER: compares two objects. Returns -1 / 0 / 1.
Overloading: same name, different params — Oracle picks by args.
Params: IN (default), OUT, IN OUT.
Invoked via dot notation: SELECT P.AGE() FROM PERSON P;

🌳 INHERITANCE

Subtypes & Polymorphism

NOT FINAL: allows subtyping. Default = FINAL.
UNDER: creates subtype. Inherits ALL attributes + methods.
NOT INSTANTIABLE: abstract type — can't be directly instantiated.
OVERRIDING: required when re-implementing a parent method.
TREAT(VALUE(p) AS SubType): downcasting. IS OF (SubType): checks membership.
Object tables are polymorphic — store supertype AND subtype instances.

💾 LOBs & VIEWS

Large Objects & Object Views

BLOB: binary (images, video). CLOB: large text (UTF-8). NCLOB: multilingual text (Unicode).
Object View: applies OO structure on top of relational tables — NO physical change. Allows old relational DB + new object interface to coexist.
CREATE VIEW V OF Type WITH OBJECT IDENTIFIER (col) AS (SELECT ... CAST(MULTISET(...) AS CollType));

⚡ TRIGGERS (ECA)

Event-Condition-Action

Event: INSERT / UPDATE [OF col] / DELETE on a table/view.
Condition: optional WHEN clause.
Action: PL/SQL block.
:OLD = values before DML. :NEW = values after. INSERT: only :NEW. DELETE: only :OLD.
Row-level (FOR EACH ROW): once per row. Statement-level: once per DML.
INSTEAD OF: on views only. RAISE_APPLICATION_ERROR(-20000...-20999,'msg'): custom error.

🔗 ASSOCIATIONS IN SQL3

1..*, *..*

1..* (Father-Son): Parent gets NT of REFs to children. Child gets REF to parent. Forward declaration needed for circular refs.
*..*: Both sides get a nested table of REFs to the other side.
Each association with an association class → split into two associations. Both ROLES must be defined.

📊 BIG DATA

5 Vs & Hadoop

5 Vs: Volume (quantity) · Velocity (speed) · Variety (structured/semi/unstructured) · Veracity (uncertainty/incompleteness) · Value (insights).
Vertical scaling: upgrade one machine (physical limit). Horizontal scaling: add more nodes (coordination overhead).
HDFS: files split into 64MB chunks, replicated 3×. NameNode = metadata. DataNodes = actual blocks.
MapReduce: Map(input) → [(k,v)] → Shuffle & Sort → Reduce((k,[v])) → result.

🚫 NOSQL

CAP · BASE · Types

CAP Theorem: Consistency · Availability · Partition tolerance — only 2 simultaneously. NoSQL: AP or CP.
BASE: Basically Available · Soft state · Eventually consistent. (vs ACID)
Key-Value: Dynamo, Riak. Document: MongoDB, CouchDB. Wide-Column: Cassandra, HBase. Graph: Neo4j, ArangoDB.
No standard query langSchema-lessHorizontal scale

🍃 MONGODB

Document Store

Hierarchy: Database → Collections → Documents (BSON) → Fields. _id is mandatory primary key.
Embedded docs: 1-to-1 or 1-to-many (fewer queries, possible redundancy). References: many-to-many (avoids duplication).
Field rules: cannot start with $, cannot contain dot (.), names must be unique.
Aggregation pipeline stages: $match → $unwind → $group → $project → $sort → $limit → $addFields → $count → $lookup → $facet.

📡 DISTRIBUTED DB

Fragmentation & Allocation

Stonebraker Transparency: 1. Data (logical + physical independence) 2. Network/Location 3. Replication 4. Fragmentation.
Horizontal frag (PHF): rows via σ. Reconstruct: ∪. Vertical frag: columns via π (keep PK). Reconstruct: ⋈. DHF: semi-join ⋉ on parent's fragment.
Allocation: Full replication (all sites) · Partial (some sites) · No replication (one site). NP-complete problem.

🗂️ ORACLE PARTITIONING

Range · List · Reference

PARTITION BY RANGE (col): PHF — splits rows by value range.
PARTITION BY LIST (col): PHF — splits by discrete value list.
PARTITION BY REFERENCE (fk): DHF — child inherits parent's partitioning via FK constraint.
